##二、版本的查看与复原 1. 场景的创立： 首先编写一个read.me的text文本，输入以下内容（记得使用vs code编辑而不是系统自带的记事本），并且把它放入的是仓库GitSourse目录下面，可以在git bash中输入cat readme.txt来直接看到文本中的内容是什么。 Git is a version control system. Git is free software.
2. 添加与提交： 1. 用命令git add告诉Git，把文件添加到仓库，具体就是先用git add readme.txt，实际上就是把文件放入缓冲区（暂存区）， 2. 然后再用命令git commit告诉Git，把文件提交到仓库，具体来说就是git commit -m "wrote a readme file",该命令实际上是实际上就是把暂存区的所有内容提交到当前分支，因为我们创建Git版本库时，Git自动为我们创建了唯一一个master分支，所以，现在，git commit就是往master分支上提交更改。-m后面输入的是本次提交的说明，可以输入任意内容，当然最好是有意义的，这样你就能从历史记录里方便地找到改动记录。git commit命令执行成功后会告诉你，1 file changed：1个文件被改动（我们新添加的readme.txt文件）；2 insertions：插入了两行内容（readme.txt有两行内容）。 3. 另外可以添加提交多个文件，例如： git add file1.txt git add file2.txt file3.txt git commit -m "add 3 files."
3. 内容的修改与查看 1. 随意修改read.txt中的内容几次后，使用git status命令可以让我们时刻掌握仓库当前的状态，输入后回车后结果会告诉我们，readme.txt被修改过了，但还没有准备提交的修改。 2. 而Untracked表明有文件还没有被添加和提交，也就是连暂存区都没有进入，那么就需要针对该文件重复git add使其进入。 3. 另外针对刚刚已经修改的文件可以再使用命令git diff readme.txt查看该文本中具体修改了什么。 “—“表示变动前的版本，”+++“表示变动后的版本。@@ -1,2 +1,2 @@ //变动前的文件从第一行开始的2行与变动后的文件第一行开始的2行存在不同。//以下是比较信息 一行前面有- + ！或者没有表示，分别表示删、增、改、没有修改。 4. 然后逐步使用命令git add readme.txt和命令git commit -m "add distributed"（-m后面表明该文件最新修改了什么）就表明提交了新的修改的文件。最后用命令用git status看看仓库的当前状态，nothing to commit, working tree clean就是没有需要提交的修改，工作目录是干净的。 5. 像这样，你不断对文件进行修改，然后不断提交修改到版本库里，就好比玩RPG游戏时，每通过一关就会自动把游戏状态存盘，如果某一关没过去，你还可以选择读取前一关的状态。有些时候，在打Boss之前，你会手动存盘，以便万一打Boss失败了，可以从最近的地方重新开始。Git也是一样，每当你觉得文件修改到一定程度的时候，就可以“保存一个快照”，这个快照在Git中被称为commit。一旦你把文件改乱了，或者误删了文件，还可以从最近的一个commit恢复，然后继续工作，而不是把几个月的工作成果全部丢失。 6. 值得注意的是git管理的是修改而不是文件本身，也就是说git commit只负责把暂存区的修改提交了，如果某些修改没有被git add命令加入暂存区，那么是无法被提交的。所以每一次的修改最好都是先git add，最后只使用一次git commit一并提交了。
4. 撤销修改 1. 命令git checkout -- readme.txt（记得–前后都有空格）意思就是，把readme.txt文件在工作区的修改全部撤销，这里有两种情况：一种是readme.txt自修改后还没有被放到暂存区，现在，撤销修改就回到和版本库一模一样的状态；一种是readme.txt已经添加到暂存区后，又作了修改，现在，撤销修改就回到添加到暂存区后的状态。总之，就是让这个文件回到最近一次git commit或git add时的状态。只要没有提交都可以撤销修改的动作。 2. 另外，命令git reset HEAD readme.txt可以把暂存区的修改撤销掉（unstage），重新放回工作区，（用HEAD时，表示最新的版本。再用git status查看一下，现在暂存区是干净的，工作区有修改。 3. 场景A当你改乱了工作区某个文件的内容，想直接丢弃工作区的修改时，用命令git checkout – file。场景B：当你不但改乱了工作区某个文件的内容，还添加到了暂存区时，想丢弃修改，分两步，第一步用命令git reset HEAD ，就回到了场景A，第二步按场景A操作。场景C：已经提交了不合适的修改到版本库时，想要撤销本次提交，参考下面的版本回退5.和6.两节，不过前提是没有推送到远程库。
5. 查看历史版本记录： 1. 使用命令git log可以查看历史记录，也就是每次commit的版本是怎样的，git log命令显示从最近到最远的提交日志 commit后面跟的是id号，用于标记每个版本，可以用命令git log --pretty=oneline仅仅查看每个id号，这样更简洁。 1. ==如何使用如果使用可视化工具查看Git历史==
6. 版本的回退 1. 回退到上一个版本是使用命令 git reset，在Git中，用HEAD表示当前版本，也就是最新的提交的版本，上一个版本就是HEAD，上上一个版本就是HEAD，当然往上100个版本写100个比较容易数不过来，所以写成HEAD~100，具体回到上一个版本的命令是git reset --hard HEAD^，可以点开文本readme.txt来查看内容已经改变，回到上一次修改前，使用git log命令查看版本库状态已经可以发现最新的版本是这个回到上一个版本的版本，也就是此时的HEAD是之前的HEAD^ 2. 若关机后第二天若想查看前面1.中被覆盖掉的最先前的最新版本可以使用命令git reflog就可以查看每一次更改的id号，使用命令git reset --hard e30fb就可以回到那个被覆盖掉的修改的版本，版本号没必要写全，前几位就可以了，Git会自动去找。
7. 文件的删除 1. 现在有以下场景，你在工作目录GitSourse中新创建一个test.txt，并且通过命令git add和git commit把它提交了，此时在输入命令rm test.txt后回车后在工作目录里面看会发现它消失了，但它实际仍然被保留在版本库中。 2. 此时有两个选择，一个是通过命令git rm test.txt和git commit -m "remove test.txt"把它从版本库中也移除，另一个选择是 通过命令git checkout -- test.txt把它又放回工作目录中去。 3. 值得注意的是：从来没有被添加到版本库就被删除的文件，是无法恢复的！

##四、分支管理 1. 简介 分支在实际中有什么用呢？假设你准备开发一个新功能，但是需要两周才能完成，第一周你写了50%的代码，如果立刻提交，由于代码还没写完，不完整的代码库会导致别人不能干活了。如果等代码全部写完再一次提交，又存在丢失每天进度的巨大风险。现在有了分支，就不用怕了。你创建了一个属于你自己的分支，别人看不到，还继续在原来的分支上正常工作，而你在自己的分支上干活，想提交就提交，直到开发完毕后，再一次性合并到原来的分支上，这样，既安全，又不影响别人工作。其他版本控制系统如SVN等都有分支管理，但是用过之后你会发现，这些版本控制系统创建和切换分支比蜗牛还慢，简直让人无法忍受，结果分支功能成了摆设，大家都不去用。但Git的分支是与众不同的，无论创建、切换和删除分支，Git在1秒钟之内就能完成！

##五、标签管理 1. 标签的作用 发布一个版本时，我们通常先在版本库中打一个标签（tag），这样，就唯一确定了打标签时刻的版本，Git的标签虽然是版本库的快照，但其实它就是指向某个commit的指针（跟分支很像对不对？但是分支可以移动，标签不能移动），所以，创建和删除标签都是瞬间完成的。而且commit的id号很长，如果使用标签就会简化很多。标签总是和某个commit挂钩。如果这个commit既出现在master分支，又出现在dev分支，那么在这两个分支上都可以看到这个标签。 2. 创建标签 1. 在Git中打标签非常简单，首先，切换到需要打标签的分支上。然后，敲命令git tag v1.0就可以打一个新标签。 2. 默认标签是打在最新提交的commit上的。有时候，如果忘了打标签，比如，现在已经是周五了，但应该在周一打的标签没有打，怎么办？方法是用git log --pretty=oneline --abbrev-commit找到历史提交的commit id（例如f52c633），然后打上git tag v0.9 f52c633。 3. 可以用命令git tag查看所有标签，但是仅仅看到的是诸如v1.0,v1.1等名；而且标签不是按时间顺序列出，而是按字母排序的。 4. 可以用git show v1.0查看标签的详细信息。 5. 可以创建带有说明的标签，用-a指定标签名，-m指定说明文字。例如：git tag -a v0.1 -m "version 0.1 released" 1094adb。用命令git show 可以看到说明文字。 3. 操作标签 1. 因为创建的标签都只存储在本地，不会自动推送到远程。所以，打错的标签可以在本地安全删除，例如：git tag -d v0.1。 2. 如果要推送某个标签到远程，使用命令git push origin ，例如：git push origin v1.0。或者，一次性推送全部尚未推送到远程的本地标签：git push origin --tags 3. 如果标签已经推送到远程，要删除远程标签就麻烦一点，先从本地删除：然后，从远程删除。删除命令也是push，但是两条代码格式如下：git tag -d v0.9;git push origin :refs/tags/v0.9。要看看是否真的从远程库删除了标签，可以登陆GitHub查看。
